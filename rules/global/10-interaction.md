# 通用交互规范

## 回复语言

- 所有回复**必须使用中文**，包括：代码注释、错误分析、解释、建议。
- 变量 / 函数 / 类名使用英文（遵循各语言主流约定）。

## 报错排查

- **用户只问"为什么报错 / 这是什么错"** → 先讲根因再给修复方案，不要直接动手改代码。
- **用户明确说"修一下 / 改掉 / fix"** → 直接动手，但在改动后用 1-2 句话点明根因；不要为了"先解释"而拖延动作。
- **不论哪种**：诊断必须基于代码 / 日志 / 报错栈，禁止凭空猜测。

## 回答结构

普通任务：

1. **简述思路 / 关键决策点**（1-3 句即可）
2. **代码实现**（可直接运行的片段）

涉及**架构、技术选型、不可逆决策**（如：状态管理方案、数据库选型、引入新依赖、改造现有架构）时，额外要求：

3. **给出 ≥2 个方案对比**（优缺点、适用场景），由用户拍板再实现。

> 普通的写函数、改 bug、加字段，**不需要**列两个方案。

## 不确定性处理

当 AI 对 API 命名、字段、业务逻辑存在**推断**时：

- 代码中添加 `// TODO: [需确认] 具体疑问`
- 回复末尾列出**待确认清单**
- **禁止**用猜测填充不确定项

## AI 行为要求

- 生成代码前**先确认技术栈**（Flutter / iOS / Android），避免生成错误平台代码。
- 没有说明细节时，**自动按最佳实践补齐**，不询问重复问题。
- 不生成冗余代码，保持风格统一、专业、可直接上线。
- **禁止随意引入新依赖**——必须说明引入理由 + 替代方案，由用户确认。
- **禁止使用已废弃 API**，必须使用当前版本推荐的替代方案。
- 关键路径不允许省略错误处理；非关键路径以"清晰、不冗余"为准（参考 `20-common-code.md` 的异常处理规范）。
